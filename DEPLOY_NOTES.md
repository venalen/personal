# Railway Deployment Notes

Notes from deploying `expense_tracker` to Railway. Reference for future services.

## Initial Setup

```sh
npm install -g @railway/cli
railway login
cd expense_tracker   # or whatever project directory
railway init         # creates a new Railway project
```

## Add a Database

From the Railway dashboard: **+New** → **Database** → **PostgreSQL**

- Railway auto-provisions a managed Postgres instance (password-protected, private network only)
- No Dockerfile or config needed for the database — Railway handles it

### Link the database to your app

In the dashboard: click your app service → **Variables** → **Add Reference Variable** → select the Postgres service → `DATABASE_URL`

This injects the connection string into your app's environment. Without this, your app won't know how to reach the database.

## Dockerfile

Railway can build from a Dockerfile or use Nixpacks. For a monorepo with separate client/server directories, a Dockerfile gives more control.

Key lessons:
- Copy all `package.json` files before running `npm install` (especially if you have `postinstall` scripts that reference subdirectories)
- Use `WORKDIR` instead of `cd` for `npm install` — npm resolves `package.json` from the working directory, not the shell's `cd` target
- Set `NODE_ENV=production` in the production stage
- Copy migration files into the production image if your app runs migrations on startup

### railway.json

When using a Dockerfile, set the builder explicitly:

```json
{
  "$schema": "https://railway.com/railway.schema.json",
  "build": {
    "builder": "DOCKERFILE"
  }
}
```

Do NOT add `startCommand` — it overrides the Dockerfile's `CMD` and will fail if the production stage doesn't have a root `package.json`.

## Deploying

```sh
railway service      # select which service to deploy to
railway up           # uploads local files and builds on Railway
```

- `railway up` deploys from **local files** (including uncommitted changes), not from git
- Ctrl-C after upload just disconnects the log stream — the build continues on Railway
- **Always run `railway service` first** to confirm you're deploying to the app, not the database. Deploying a Dockerfile to the Postgres service will crash it.

### If you accidentally deploy to the database

In the dashboard: click the Postgres service → remove the bad deployment → the default managed image will restart automatically.

### Monitoring

```sh
railway logs         # stream logs from your terminal
```

Or: dashboard → service → Deployments → click a deployment

## Networking

### Private networking (default, no egress fees)

Services communicate over Railway's private network using internal hostnames. The `DATABASE_URL` reference variable uses this by default.

- Both services must have **Private Networking** enabled (Settings → Networking)
- If the database isn't reachable, check that private networking is on for both services

### Public domain

Not generated by default. To expose your app to the internet:

Dashboard → app service → **Settings** → **Networking** → **Public Networking** → **Generate Domain**

Gives you a `*.up.railway.app` URL. The domain is free; you pay for compute and egress usage.

### DATABASE_PUBLIC_URL

Avoid using this — it routes traffic over the public internet and incurs egress fees. Stick with the private `DATABASE_URL`.

## Database Connection Tips

### Startup timing

The database may not be ready when your app starts. Add retry logic to your startup:

```ts
async function waitForDb(retries = 10, delay = 3000) {
  for (let i = 0; i < retries; i++) {
    try {
      await migrate();
      return;
    } catch (err) {
      if (i === retries - 1) throw err;
      console.log(`Database not ready, retrying in ${delay / 1000}s... (${i + 1}/${retries})`);
      await new Promise((r) => setTimeout(r, delay));
    }
  }
}
```

### SSL

Railway Postgres may require SSL. Handle both local and production:

```ts
const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  ...(process.env.NODE_ENV === 'production' ? { ssl: { rejectUnauthorized: false } } : {}),
});
```

## Costs

Railway is usage-based. For a small personal app, expect a few dollars/month. Set a spending limit in account settings to avoid surprises.

## Auto-deploy from GitHub (optional)

Instead of `railway up`, connect your repo in the dashboard: service → **Settings** → **Source**. Pushes to your branch will trigger automatic deploys. Only committed/pushed code gets deployed (unlike `railway up` which uses local files).
